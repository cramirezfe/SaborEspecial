# Production Deploy Guide — Sabor Especial

This guide takes you from a working local instance to a live URL you can hand to a cafetería. For getting a local instance running first, see **README.md**.

---

## What you need before starting

- A working local instance (`npm run setup` + `npm start` passes the health check)
- A [Vercel](https://vercel.com) account (free tier works)
- The Vercel CLI installed (`npm i -g vercel`)
- Your Supabase credentials from `npm run setup` (they're in `.env.local`)

---

## Step 1 — Link the repo to Vercel

```bash
vercel link
```

Follow the prompts: log in, select your team/org, and link to a new project. This creates a `.vercel/` folder locally — don't commit it.

---

## Step 2 — Set environment variables in Vercel

Your `.env.local` has every variable the app needs. Copy them into Vercel one by one:

```bash
vercel env add SUPABASE_URL
vercel env add SUPABASE_ANON_KEY
vercel env add SUPABASE_SERVICE_ROLE_KEY
vercel env add DATABASE_URL
vercel env add APP_BASE_URL          # e.g. https://mi-soda.vercel.app
vercel env add ADMIN_SECRET          # from your .env.local — share only with admin
vercel env add ORDERS_PASSWORD       # from your .env.local — share only with kitchen
vercel env add CORS_ORIGIN           # exact frontend origin, e.g. https://mi-soda.vercel.app
vercel env add RESEND_API_KEY        # see Step 5; add an empty value for now if skipping
```

Each command prompts for the value and asks which environments to apply it to (Production, Preview, Development). Select **Production** (and Preview if you want deploy previews to work).

> `ADMIN_SECRET` and `ORDERS_PASSWORD` were generated by `npm run setup` and printed at the end. They're also in `.env.local` if you need to look them up.

---

## Step 3 — Deploy to production

```bash
vercel --prod
```

Vercel builds and deploys. When it finishes it prints your live URL, e.g. `https://mi-soda.vercel.app`.

Verify the API is up:

```bash
curl https://mi-soda.vercel.app/api/health
# Expected: {"status":"ok"}
```

If it returns an error, check **Vercel → your project → Logs** for the specific failure.

---

## Step 4 — Onboard the cafetería as a tenant

Every cafetería needs a row in the database. The local seed created a `test-cafeteria` — for production you create a real one:

```bash
curl -X POST https://mi-soda.vercel.app/api/onboard \
  -H "Content-Type: application/json" \
  -d '{
    "slug": "mi-soda",
    "name": "Mi Soda",
    "adminEmail": "admin@mi-soda.com"
  }'
```

Expected response:
```json
{"cafeteriaId": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx", "slug": "mi-soda"}
```

This creates the cafetería row and default settings: 15 max meals, sales window 10:00–12:00 Costa Rica time, timezone `America/Costa_Rica`. The admin can change these from the management panel after launch.

Verify it worked:

```bash
curl "https://mi-soda.vercel.app/api/dashboard?slug=mi-soda"
# Expected: JSON with cafetería data, not "cafeteria not found"
```

## Step 5 — Set up email notifications (Resend)

Order confirmations and payment notifications go through [Resend](https://resend.com). Skip this step if you don't need email — orders still work and failed email calls are logged silently to the `error_logs` table.

1. Create a free account at [resend.com](https://resend.com).
2. Go to **Domains → Add Domain** and verify your sender domain (e.g. `notificaciones@mi-soda.com`). Follow the DNS instructions — usually takes a few minutes.
3. Go to **API Keys → Create API Key**.
4. Add it to Vercel and redeploy:

```bash
vercel env add RESEND_API_KEY
vercel --prod
```

5. Test by placing a real order at `/s/mi-soda` with a valid email address and confirming the confirmation email arrives.

---

## Step 6 — (Optional) Add a custom domain

`mi-soda.vercel.app` works, but a custom domain looks more professional.

1. In Vercel → your project → **Settings → Domains**, click **Add**.
2. Enter your domain (e.g. `pedidos.mi-soda.com`).
3. Follow the DNS instructions Vercel shows — add a CNAME or A record at your registrar.
4. Wait for DNS propagation (usually under 10 minutes).

No code changes needed. The app derives its API base URL from `window.location.origin` automatically.

After adding the domain, update `APP_BASE_URL` and `CORS_ORIGIN` to match, then redeploy:

```bash
vercel env rm APP_BASE_URL && vercel env add APP_BASE_URL   # enter new domain
vercel env rm CORS_ORIGIN  && vercel env add CORS_ORIGIN    # enter new domain
vercel --prod
```

## Step 7 — Run the test suite against the live URL

```bash
API_BASE_URL=https://mi-soda.vercel.app npm test
```

Suites that need extra env vars (multi-tenant tokens) skip gracefully and don't fail the run. A clean baseline:

```
✅ Suite 1 — Midnight Reset Logic
✅ Suite 2 — Concurrency & Race Conditions
✅ Suite 3 — Multi-Tenant Isolation (skipped — tokens not set)
✅ Suite 4 — High-Latency Resilience
✅ Suite 5 — Input Validation
✅ Suite 6 — Offline-First PWA
```

## Step 8 — Manual checks before the first lunch service

No automated coverage for these — walk through each one:

| Area | How to test |
|------|-------------|
| SINPE payment flow | Place an order with SINPE → confirm the confirmation message shows the correct phone number |
| Credits / packages | Buy a package at `/packages` → spend a credit on an order → verify balance updates |
| Excel export | Click `Exportar Excel` in the admin panel → open the CSV and check columns and data |
| Email notifications | Place a test order with a real email → confirm the confirmation email arrives |
| Offline fallback | Load the app → DevTools → Network → Offline → reload → last snapshot should appear |
| Slow 3G double-submit | DevTools → Network → Slow 3G → submit order → confirm no duplicate on double-click |
| Midnight counter reset | Around 06:00 UTC (midnight Costa Rica), verify the available-meal counter resets to the daily max |
| Multi-tenant isolation | Create a second cafetería → confirm its orders don't appear in the first cafetería's admin panel |

See `PRODUCTION_CHECKLIST.md` for the complete pre-launch validation checklist.

## Step 9 — Share URLs with staff

Three links to bookmark and distribute:

| Who | URL | Password |
|-----|-----|----------|
| Customers | `https://your-domain.com/s/mi-soda` | None (safe to post publicly) |
| Admin / owner | `https://your-domain.com/management.html?slug=mi-soda` | `ADMIN_SECRET` |
| Kitchen staff | `https://your-domain.com/deliveries.html?slug=mi-soda` | `ORDERS_PASSWORD` |

The customer URL is safe to share via WhatsApp, a notice board, etc.

---

## Step 10 — Monitor the first lunch rush (10:00–12:00 CR)

Watch these four signals during the first service:

| Signal | Where to check | Action if wrong |
|--------|---------------|-----------------|
| `availableMeals` goes negative | Supabase → `orders` table row count vs `settings.max_meals` | Check the `place_order_atomic` function is deployed |
| Orders not showing in kitchen | Deliveries page auto-refreshes every 30 s — check the network tab | Confirm `ORDERS_PASSWORD` is set correctly |
| Duplicate orders | Supabase → `orders` table, same name same minute | Check browser console for JS errors in `app.js` |
| Emails not arriving | Resend dashboard → Logs | Verify `RESEND_API_KEY` and sender domain are confirmed |

Supabase logs: **Dashboard → Logs → API** — real-time view of every request and error.
Vercel logs: **Vercel → project → Deployments → Functions** tab.

| # | Done when… |
|---|------------|

| 5 | `PRODUCTION_CHECKLIST.md` is fully green |
| 6 | First 30 minutes of lunch service has no errors in Vercel or Supabase logs |

## Common issues

**`cafeteria not found` on API calls**
The slug in the URL doesn't match any row in the `cafeterias` table. Check you ran Step 4 with the same slug you're using in the URL.

**`invalid API key` from Supabase**
Anon and service_role keys are swapped. The backend uses `SUPABASE_SERVICE_ROLE_KEY`; the browser gets the anon key automatically from `/config.js`. Double-check the values in Vercel → Environment Variables.

**Order count doesn't reset at midnight**
The reset is timezone-aware (`America/Costa_Rica`, UTC-6). If counts don't reset, confirm Supabase isn't filtering by raw UTC dates — `getDayKey()` in `app.js` handles the conversion.

**Service Worker serving stale pages after a redeploy**
Bump the cache version constant in `sw.js`, commit, and redeploy (`vercel --prod`). Clients pick up the new version on their next page load.
